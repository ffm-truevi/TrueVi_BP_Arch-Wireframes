TrueVi Library — API Contract

The wire protocol between the Page Renderer and the TrueVi Library. Five endpoints, eight canonical schemas, one taxonomy of activity events. Stable across renderer evolution, content backend swaps, and inference engine replacements.

One sentence

The renderer asks the Library to compose a page or answer a question and reports back what the user did. Everything else lives behind that wall.

1 · Overview

MethodPathPurpose
POST/library/compose-pageRender a page composition for a given user + route
POST/library/ask-anythingGenerate or retrieve a Search Response for an Ask Anything query
POST/library/eventsIngest one or more activity events (fire-and-forget)
GET/library/inference/:userIdRead derived signals for a user (admin/debug only)
GET/library/healthLiveness probe

2 · Conventions

  • Transport. HTTPS only. HTTP/2.
  • Encoding. JSON, UTF-8. Content-Type: application/json.
  • Time. RFC 3339, always UTC: 2026-05-11T14:32:00Z.
  • IDs. UUIDv7 for entities (sortable by creation time). Slug strings for human-readable taxonomies (ministry, subject, series).
  • Casing. camelCase on the wire.
  • Null vs missing. Optional fields are omitted, not nulled. The wire format favors smaller payloads.
  • Booleans. Always lowercase true/false. Never 0/1.
  • Locale. Every user-visible string is locale-keyed: { en: "...", es: "..." }. EN is the fallback.
  • Pagination. Cursor-based: nextCursor field. No offset pagination.
  • Backwards compatibility. Additive only. Removing or renaming fields is a major version change (§23).

3 · Authentication

All endpoints accept an internal service token issued by the TrueVi platform. The Page Renderer is the sole authorized client; the Library is not directly callable from end-user browsers. 

Authorization: Bearer <service-token>
X-TrueVi-Renderer-Version: 2026.05.11
X-Request-Id: 01919ab0-f8e2-7d34-a3c1-9e2f1c8b5d4a
  • Authorization — required; rejected with 401 if missing or invalid
  • X-TrueVi-Renderer-Version — required; used for Library-side compatibility logic
  • X-Request-Id — recommended; echoed back in responses for trace correlation

4 · Error Format

All errors return a stable shape with a typed code and a human-readable detail.

HTTP 4xx | 5xx
{
  "error": {
    "code":     "INVALID_PAGE_TYPE",
    "message":  "Unknown pageType: 'profile'",
    "requestId": "01919ab0-...",
    "retryable": false,
    "details":  { "pageType": "profile" }
  }
}
HTTPcodeMeaning
400BAD_REQUESTMalformed JSON or schema violation
400INVALID_PAGE_TYPEpageType not recognized
401UNAUTHORIZEDMissing or invalid service token
404NOT_FOUNDResource (route, response hash, user) does not exist
409EXPERIMENT_CONFLICTForced variant does not match user's existing assignment
422NO_TEMPLATE_FOR_PAGE_TYPENo published template exists for this pageType (Library bug or unpublished setup)
429RATE_LIMITEDRenderer exceeded rate; retry-after header included
500INTERNALUnexpected server error; safe to retry
503DEGRADEDLibrary running in fallback mode; response may lack personalization

5 · Identity

Every request carries an identity envelope. The Library uses it to look up inferences, allocate experiments, and write activity events.

Identity {
  "userId":       "01919ab0-...",   // null until signup
  "anonymousId":  "01919ab0-...",   // required, set on first visit
  "sessionId":    "01919ab0-...",   // required, rotates per session
  "deviceId":     "01919ab0-..."    // optional, localStorage
}

Identity precedence. When multiple IDs are present, the Library scopes inferences by userId if present, else anonymousId. sessionId is used only for session-level caching and analytics. deviceId is used for cold-start defaults.

The canonical shape for everything the Library returns as "content." Used uniformly across video, series, ministry, subject, and scripture types.

    "slug":       "mercy-hill",
    "name":       "Mercy Hill Church",
    "logoUrl":    "https://...",
    "pastorName": "Jane Doe"
  },
  "series": {                                              // optional
    "slug":           "forgiveness-series",
    "episodeNumber":  3,
    "totalEpisodes":  7

  // taxonomy
  "subjects": [

    { "tag": "grief-loss", "relevance": 0.3 }

  // theological metadata
  "denomination":        "nondenominational",
  "theologicalScore":    0.92,                       // 0..1, guardrail filter floor = 0.75
  "contentWarnings":     [],
  "pastoralReviewStatus": "approved",                  // approved | pending | flagged

  // engagement metrics — aggregate, public

    "views":           12450,

    "watchProgress":  0.42,             // 0..1; 1 = completed

    "completedAt":    null,

9 · FilledInstance

What the Library returns inside a composition. The renderer takes this and hydrates the component directly.

FilledInstance {
  "instanceId":    "rail-continue-watching-001",
  "componentType": "ContentRail",
  "version":       "1.4.2",

  "resolvedConfig": { ... },                       // merged with Library overrides
  "content":        [ ContentItem, ContentItem, ... ],
  "copy": {
    "title":    "Continue Watching",
    "subtitle": null,
    "cta":      "See all"
  },
  "why": WhyPayload,                              // see §13
  "experimentVariant": "home-v3-variant-B",
  "ttl": 900                              // seconds — per-instance override
}

10 · SlotSpec

The template-level shape of a slot. Stored on the template; never created at render time.

SlotSpec {
  "slotId":        "dynamic-rails",
  "kind":          "FIXED" | "DYNAMIC",

  // FIXED slots — admin pinned a specific component
  "component":     "ContentRail",                  // FIXED only
  "instance":      ComponentInstance,                // FIXED only — see §8

  // DYNAMIC slots — admin defined constraints, Library decides
  "allows":        ["ContentRail", "MinistryRail"],   // DYNAMIC only
  "minCount":      3,                                // DYNAMIC only
  "maxCount":      10,                               // DYNAMIC only

  // shared
  "tierVisibility": ["new", "vine", "plus"]
}

12 · ExperimentAssignment

14 · POST /library/compose-page

POST/library/compose-page

Returns the full composition for a page request. One call per page render.

Request

{
  "pageType":    "home",
  "route":       "/",
  "routeParams": {},                            // e.g. { ministrySlug: "mercy-hill" }

  "identity":    Identity,
  "request":     RequestEnvelope,

  "overrides": {                                 // optional, QA-only
    "forceTemplateId": null,
    "forceVariant":    null,
    "viewAsCohort":    null                     // "first-visit" | "power" | { userId }
  }
}

Response (200)

{
  "templateId":       "home-v3",
  "templateVersion":  "1.4.2",
  "pageType":         "home",
  "experimentAssignments": [ ExperimentAssignment, ... ],

  "composition": {
    "slots": [
      {
        "slotId":    "hero",
        "kind":      "FIXED",
        "instances": [ FilledInstance ]
      },
      {
        "slotId":    "dynamic-rails",
        "kind":      "DYNAMIC",
        "instances": [ FilledInstance, FilledInstance, ... ]
      }
    ]
  },

  "cacheKey":        "home:user-abc:v3:...",
  "ttl":             900,
  "serverTime":      "2026-05-11T14:32:00Z",
  "degraded":        false                 // true when Library is in fallback mode
}

Behavior notes

  • The renderer must respect kind. A FIXED slot will always have exactly one instance. A DYNAMIC slot will have between minCount and maxCount instances (or zero if Library determined nothing fits — see fallback rules in architecture §6).
  • The order of slots in the response is the render order. Do not reorder.
  • Within a DYNAMIC slot, the order of instances is the render order. The Library is responsible for prioritization.
  • When degraded: true, the renderer should display normally — the Library has returned editorial defaults rather than personalized output.

15 · POST /library/ask-anything

POST/library/ask-anything

Submit a free-form question. Returns a Search Response composition, optionally retrieved from the semantic cache.

Request

{
  "query":      "How do I forgive my father?",
  "identity":   Identity,
  "request":    RequestEnvelope,

  "context": {                                // optional — for multi-turn
    "previousResponseId": null,
    "conversationId":    null
  }
}

Response (200)

{
  "responseId":     "a3f9c2...",                // semantic hash
  "responseUrl":    "/answer/a3f9c2...",
  "templateId":     "search-response-empathic-v1",
  "composition":    { "slots": [ ... ] },        // same shape as compose-page

  "meta": {
    "cached":        true,
    "generatedAt":   "2026-05-09T...",
    "semanticHash":  "a3f9c2...",
    "intent":        "emotional-pain",
    "shareableUrl":  "https://truevi.tv/answer/a3f9c2..."
  }
}

Notes

  • Queries are normalized and hashed semantically — equivalent queries return identical responses (architecture §15).
  • The composition is returned in the same shape as compose-page so the renderer hydrates it identically.
  • The Library may select a different templateId per query — the AI picks the right Search Response template (empathic, theological, practical, scripture-heavy) for the intent.

16 · POST /library/events

POST/library/events

Ingest activity events. Batched. Fire-and-forget. The renderer does not block on responses.

Request

{
  "identity": Identity,
  "events": [
    {
      "eventType":             "video_completed",
      "timestamp":             "2026-05-11T14:32:45.231Z",
      "slotId":               "dynamic-rails",
      "componentInstanceId":   "rail-continue-watching-001",
      "contentId":             "01919ab0-...",
      "experimentAssignments": [ ExperimentAssignment, ... ],
      "payload": {
        "durationSec":     1845,
        "watchedSec":      1820,
        "completionPct":   0.99
      }
    },
    { ... },
    { ... }
  ]
}

Response (202 Accepted)

{ "accepted": 3, "rejected": 0 }

Behavior notes

  • The endpoint always returns 202 Accepted on a well-formed batch — never 200. Processing is asynchronous.
  • Rejected events are dropped silently; the renderer is not expected to retry. Lost events are acceptable.
  • Batch size soft-limit: 50 events. Hard-limit: 200. Larger batches are split or partial-rejected.
  • The browser may send events via navigator.sendBeacon on page unload — the endpoint must accept those payloads.

17 · GET /library/inference/:userId

GET/library/inference/{userId}

Read derived signals for a user. Admin/debug only — used by Console "View as User" mode and the inference inspector.

Response (200)

DerivedSignals  // see §11

Returns 404 if no signals exist for the user yet (e.g., pre-first-event). Caller should handle as a cold-start case.

21 · Caching & ETags

The Library uses both HTTP-level and application-level caching.

HeaderUsed byPurpose
Cache-Controlcompose-page, ask-anythingprivate, max-age=<ttl> — respects per-user TTL
ETagcompose-pageComposition fingerprint; renderer may send If-None-Match on revisit
Varycompose-pageAlways includes X-TrueVi-Identity so CDN doesn't cross-bleed
Agecompose-pageSeconds since the composition was generated

304 Not Modified

The renderer is encouraged to send If-None-Match when re-fetching a page. The Library returns 304 if the composition would be identical, saving the full payload.

22 · Rate Limiting

The Library enforces per-renderer rate limits.

EndpointLimit (per renderer instance)
compose-page200 req/s sustained · 500 burst
ask-anything20 req/s sustained · 50 burst
events2000 req/s sustained · 5000 burst
inference10 req/s sustained · 20 burst

When exceeded, Library returns 429 with a Retry-After header. The renderer should fall back to the most recent cached composition if available; otherwise render a degraded page from editorial defaults.

23 · Versioning

The API is versioned via the URL path and the renderer header. Current major version: v1. 

https://library.truevi.internal/v1/compose-page
X-TrueVi-Renderer-Version: 2026.05.11
  • Minor changes (additive fields, new event types, new pageTypes) — no version bump; renderers ignore unknown fields gracefully.
  • Major changes (removing fields, changing field semantics, restructuring composition shape) — new path prefix (/v2); both versions served in parallel until migration is complete.
  • Renderer-version header — the Library uses this to apply compatibility shims for older renderers in production while a new version is deploying.
