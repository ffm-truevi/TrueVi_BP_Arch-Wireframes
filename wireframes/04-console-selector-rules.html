Template Selector Rules

Admin-authored rules that decide which template fires for which user, per page type. Priority-ordered. First match wins. A default template always exists as the fallback. This is where the Library learns the admin's intent for how templates should be deployed.

One sentence

Selector rules are opinionated routing over the user context. Experiments override rules; rules override the default template. See M1.1 §8 for the selection precedence.

Annotated walkthrough

1
Page type tabs Selector rules are scoped per page type. Each tab shows a count of active rules. Clicking a tab swaps the rule list below.
2
Priority column Lower priority value = evaluated first. Drag the grip handle to reorder. Reordering visually rewrites the priority integer in 10-unit increments (#10, #20, #30, …) so new rules can be inserted between existing ones without renumbering everything.
3
Condition shorthand The rule row shows the human-readable condition. The full predicate lives in the rule's condition JSON column. The editor panel translates the JSON to and from the builder UI.
4
The default rule is required Every page type has a non-deletable default template that fires when no rule matches. The default status toggle is locked on. Admins can swap which template is the default but cannot remove it. The Library returns 422 NO_TEMPLATE_FOR_PAGE_TYPE if a default is missing (M1.2 §4).
5
Test this rule Pick a synthetic cohort and evaluate. The system runs the entire rule chain against the cohort and reports which rule would win plus why earlier rules did not. Critical for catching ordering mistakes.

Condition predicates

Conditions are evaluated against the merged user context (DerivedSignals + RequestEnvelope from M1.2 §5–6 + §11). All fields available in the picker:

FieldTypeSource
visitorTypeenumfirst-visit · returning · regular · power
tierenumnew · vine · plus
sessionCountintFrom identities table
inferredFaithStageenumFrom inferences.inferred — confidence threshold applied
inferredLifeStageenumFrom inferences.inferred
inferredLifeEventssetUsed with in / not in
deviceClassenumdesktop · tablet · mobile · tv
localestringe.g. en-US
geoCountryISO-3166-1From IP, country-level only
referrerTypeenumdirect · search · social · partner
utmCampaignstringFrom request envelope
lastActiveMinistrystringSlug
watchCompletionRatenumeric0..1

Operators: ==, !=, >, >=, <, <=, in, not in, matches (regex).

Multiple conditions on the same rule are joined with AND. OR is achieved by creating multiple rules at the same priority — both will match independently; the first written wins.

Interactions

Reordering by priority

Drag the grip handle vertically. The dragged rule animates to the new slot. On drop, all affected rules' priority values are rewritten in the 10-unit grid. Persisted on drop.

Deleting a rule

Confirmed via the ⋯ menu. Selector rules are hard-deleted (no soft delete). Deleting the default is blocked — admin must first promote another rule to default.

Promoting a rule to default

Right-click ⋯ → "Make default." Swaps the rule's pinned position. The previous default becomes a regular high-priority rule.

Interaction with experiments

Selector rules are bypassed entirely when a user is enrolled in a live experiment for the same page type. The experiment variant overrides any rule decision (M1.1 §11). The UI shows a banner on the relevant tab: "2 active experiments override these rules for some users."
