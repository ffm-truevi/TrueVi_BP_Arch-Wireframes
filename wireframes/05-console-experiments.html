Experiments

Author, run, monitor, and conclude A/B experiments. Each experiment pits two or more templates of the same page type against each other. Stable cohort assignment, four locked primary KPIs, automatic guardrail monitoring, statistical-significance readout, one-click promote-winner.

One sentence

Experiments are how the team learns what actually works. They override selector rules for assigned users and report every event with their experimentId + variantId so the inference engine can learn per-variant (M1.1 §11).

Annotated walkthrough

1
Status filters Five mutually exclusive states: live, draft, paused, concluded, archived (not shown). Concluded experiments retain their data for historical review.
2
Primary KPI column Always the single KPI that determines the winner. Locked to the four metrics from M1.1 §11: watch completion · return visit (7d) · signup conversion · Ask Anything engagement. Other KPIs are tracked but not decisive.
3
Promote button Visible only when significance ≥ threshold AND sample ≥ minimum AND no guardrail tripped. One click moves the winning variant's template into the default-template slot for that page type, archives the experiment, and reroutes all traffic.
4
Paused with guardrail tripped When a guardrail KPI (theological complaint rate, crash rate, p95 latency, bounce rate) regresses past its threshold, the system auto-pauses and emails the owner. Resume requires explicit acknowledgment.
5
Significance bar Visualizes p-value for the primary KPI against the experiment's threshold. The bar turns green when the experiment is ready to call. Hover for full statistical detail (confidence interval, effect size, sample power).
6
Conclude without winner For experiments that ran their max duration without significance. Archives the experiment, traffic reverts to selector rules + default template. Available even mid-run for kills.

Stable — same user, same variantexperiment_assignments table (M1.3 §11) with unique (experiment_id, identity_id)
Deterministic for new usersHash of (assignmentKey, experimentId) mod 100 against the allocation table
Sticky across sessionsAssignment persists in the table; sessionId rotation does not affect it

Edge cases

Two live experiments on the same page type

Both run, both assign. If a user is in both, both variants' template IDs would conflict. The system resolves this with an experiment priority — older experiments win until concluded. The newer experiment skips users already in the older one. Surfaced in the UI as "will not enroll users already in: …"

Template archived mid-experiment

Blocked at the template archive action. Archive UI shows: "Template is referenced by live experiment X. Conclude or remove from experiment first."

Allocation change mid-experiment

Not allowed once status=live. Pause first, edit, resume. Existing assignments are not redistributed when allocation changes — only future unassigned users see the new ratio.

Guardrail auto-pause

Checked every 15 minutes. If a guardrail KPI is statistically worse than control with sig > 0.90, the experiment moves to paused automatically. Email to owner. Resume requires explicit "I acknowledge the regression" checkbox.
