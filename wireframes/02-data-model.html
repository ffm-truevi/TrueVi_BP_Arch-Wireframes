TrueVi Data Model

The persistent shapes that back the architecture (M1.1) and serve the API (M1.2). Seventeen tables, four logical domains, one migration plan from the current SQLite-backed Console.

One sentence

Authoring shapes are normalized; behavioral shapes are append-only; content shapes are mirrored from the CMS; and inferences are derived projections kept in their own store so they can be regenerated from raw events at any time.

3 · Conventions

  • Primary keys — UUIDv7 (sortable by creation time). Single-column. Never composite.
  • Foreign keys — always indexed. Cascade on delete only for owned children (e.g., experiment_assignments cascade with experiments).
  • JSON columns — used for shapes that evolve (template slots, component config, event payloads). Validated at application layer; storage is type-loose.
  • Timestamps — every table has created_at and updated_at. RFC 3339 UTC. Default now().
  • Soft delete — authoring tables (templates, components, etc.) carry archived_at; never hard-deleted. Behavioral tables are hard-deleted on data subject deletion request.
  • Status field — authoring tables use status enum('draft','published','archived').
  • Versioning — templates and components have explicit version columns (semver-ish strings). Multiple versions coexist; only one is "published" at a time.
  • Naming — table names are snake_case plural. Column names snake_case singular.

4 · Storage Strategy

DomainEngineWhy
Authoring Postgres (Cloud SQL) Low-volume, transactional, foreign-key heavy. Already in the GCP plan.
Experimentation Postgres Same row shape as authoring; reuse the cluster.
Content Postgres + Vector index Postgres for the row, an attached vector store (pgvector or external) for semantic embeddings.
Activity events Append-only event store (BigQuery or Postgres partitioned) Volume can exceed 10M rows/day at scale. Time-partitioned. Never UPDATE, only INSERT.
Inferences Postgres + Redis hot cache Postgres is the durable copy; Redis serves reads inside the request path.
Search responses Postgres (hot) + Object storage (cold) Recent and shared responses in Postgres; long-cold responses serialize to GCS and regenerate on demand.

Single Postgres instance for Phase 1

Phase 1 ships everything in one Postgres database. Activity events live in a partitioned table. BigQuery moves in when volume justifies it (likely Phase 3). The schema is designed so the move requires no application-layer rewrites — only the event-write path changes.

22 · Migration Plan

The current codebase uses SQLite for the Presentation Console config and mock JSON for content. Migration is staged so the live POC keeps working at every step.

PhaseActionRisk
P1 Stand up Postgres in GCP per the existing infra plan. Create all 17 tables in truevi_library schema. Empty. Low — net-new database, parallel to SQLite
P2 Backfill: import mock-ministries.json → ministries, mock-series.json → series, mock-content.json → content_items. Generate slugs, validate FKs. Low — read-only mocks become read-only seeds
P3 Author the v1 templates that mirror today's pages. Insert into templates. Wire up routes for current URL patterns. Medium — first authoring path; needs Console builder (M2)
P4 Build the Library service. Implement compose-page using P3 templates + P2 content. Renderer reads from Library behind a feature flag, with fallback to current page code. Medium — net-new service, gated rollout
P5 Wire activity event ingestion. Implement identities + content_state materialization. Card-state badges start working off the new tables. Medium — write path goes live
P6 Build warm-tier inference. Populate inferences. Compose-page begins using personalization signals. Higher — inference quality matters; A/B against P4 baseline
P7 Cut over fully: feature flag default flips to Library-served pages. Old page code becomes fallback. Console (M2) is the authoring path. High but reversible — flag can flip back instantly
P8 Backfill remaining: experiments, editorial overrides, search responses, vector embeddings. Low

The SQLite ConsoleConfig stays — for now

Today's Presentation Console writes to a SQLite-backed console_config structure. This system stays intact during P1-P5; it becomes redundant once Console v2 (M2) is live and pointed at Postgres. The deprecation cutover is part of P7.
