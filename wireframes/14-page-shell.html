Page Shell — NavBar & Footer

The global chrome that wraps every page on the platform. NavBar at the top, Footer at the bottom, template content in between. Authored once, rendered everywhere, editable in the Management Console. Not a template — a layer above templates.

One sentence

NavBar and Footer are page shell — universal chrome that wraps the templated body of every page. Authored in one place, rendered everywhere, never duplicated into templates.

Where the shell fits in the rendering model

Until this doc, templates were treated as the entire page. They are not. Every page actually has three layers: the shell wraps the template body. 

┌─────────────────────────────────────────────────────────┐
│  NavBar      ← Page Shell · global · always present     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Template body   ← M3.2-M3.7 + M3.9 · per route         │
│  (Home, Ministry, Search Response, Series, etc.)        │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  Footer      ← Page Shell · global · always present     │
└─────────────────────────────────────────────────────────┘

This model aligns with how every modern CMS handles global chrome (Webflow, Builder.io, Sanity, Contentful).

Annotated walkthrough

1
NavBar · global, sticky Always present at the top of every page. Contains the wordmark, primary navigation, search affordance, account state, and the primary CTA. Sticks to the viewport top on scroll. Auth state (signed-in / out) swaps the right-side elements. Tier-aware: free-tier users see "Get guidance" CTA; subscribed users see avatar + tier badge.
2
Template body · slot composition The renderer calls POST /library/compose-page based on the route, gets back the template composition, and renders the slots into this region. The shell wraps but never enters this region. Templates do not need to author NavBar or Footer slots — they're handled outside.
3
Footer · global, multi-column Always present at the bottom of every page. Three columns of links plus brand block, with a bottom bar for copyright, privacy, terms. Editorial content — admin-authored, not personalized. Tier-aware: "Subscription tiers" link only appears for free users; "Account settings" replaces it for subscribers.

Tier-aware shell rules

The shell adapts to the viewing user's tier the same way components do. Common patterns:

ElementFree (new)VinePlus
"My Journey" nav linkHiddenVisibleVisible
"Sign in" link (when signed out)Visiblen/a (signed in)n/a (signed in)
"Get guidance" CTAVisibleHiddenHidden
Account avatarVisible if signed inVisibleVisible (with Plus badge)
Notification dotHiddenVisibleVisible
Footer "Subscription tiers" linkVisibleHidden (replaced by "Manage subscription")Hidden (replaced by "Manage subscription")

When monetizationMode = "open-all" globally, tier visibility rules still save but enforce only on auth-state, not paid-tier checks. The "Get guidance" CTA still hides for signed-in users; it just doesn't enforce tier subscription state.

Per-page shell suppression

In rare cases, a template needs to suppress part of the shell. The primary use case: embed views and kiosk modes. Templates can declare suppression on a per-instance basis. 

{
  "templateId": "embed-video-v1",
  "shellOverride": {
    "hideNavBar": true,
    "hideFooter": true
  },
  "slots": [ ... ]
}

Examples of templates that suppress:

  • Embed-Video — when TrueVi videos are embedded on external partner sites; only the player renders
  • Kiosk-mode — full-screen experience for church-foyer displays (future)
  • Print view — a sermon's text representation, no chrome (future)

By default no template suppresses anything. Suppression is opt-in, per template.

API integration

Two endpoints support the shell:

EndpointPurpose
GET /library/shell Called by the Renderer once per session to fetch the published shell config. Returns the full page_shell record with tier-filtered nav links and footer links already removed per the current user's tier. Cached aggressively (Edge-cacheable for signed-out traffic).
GET /console/shell · PATCH /console/shell · POST /console/shell/publish Admin endpoints for the Console editor. Same draft → publish flow as templates.

The shell is not part of the compose-page response. It is fetched separately and cached. This keeps every template render fast even when the shell config is large.

Edge cases

Shell config fails to load

Renderer falls back to a hardcoded baseline shell — wordmark only on top, copyright + privacy + terms in a minimal footer. Page still renders.

User in mid-experiment, shell changed during their session

Shell version is captured per session. If the shell is republished mid-session, the user keeps their original shell until next page load (or until session expiry). Prevents jarring layout shifts.

Locale not supported

Per the global locale settings (M2.4), unsupported locales fall back to en automatically. The shell editor warns admins when they publish without translations for all supported locales.

Template suppresses NavBar but page is shareable

Embed templates that hide NavBar must include their own minimal navigation (typically a "Powered by TrueVi" attribution with a link back to the canonical page). Enforced at publish-time validation.

Why not just put NavBar and Footer in every template?

The case for keeping shell separate

We considered making NavBar and Footer mandatory FIXED slots in every template. Rejected because:

  • Duplication. Twenty-plus templates would each have to author identical nav and footer slots.
  • Drift. An admin updating nav links would have to remember every template — guaranteed inconsistency.
  • Performance. Shell can be cached aggressively (mostly static); per-template caching would re-fetch it constantly.
  • Mental model. Templates are about a page type's content. Chrome belongs above.
  • Industry norm. Every modern CMS treats chrome as a separate layer for the same reasons.

Net: the shell pattern is unambiguously the right architecture. Templates stay focused on what they're for; the shell is a single source of truth.
