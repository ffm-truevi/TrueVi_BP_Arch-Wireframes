Template Builder

The drag-and-drop authoring surface where admins compose page templates from generic components. Three panels: palette of components, canvas of slots, properties of the selected slot. This is the hero screen of the entire system.

One sentence

The admin assembles a shape (slots + constraints) and publishes it. The Library decides what fills the shape at runtime (M1.1 §3).

Annotated walkthrough

1
Breadcrumb & status The user navigated from /console/templates to this template. Status is Draft — changes are autosaved every ~10 seconds. Only the Published version is served to users; drafts are invisible until promoted.
2
View as · cohort selector Critical preview affordance. Renders the canvas as if the selected cohort were viewing it — selector rules and inferences are evaluated against the simulated user. Without this, you can't QA the personalization layer. Backed by POST /library/compose-page with the overrides.viewAsCohort field (M1.2 §14).
3
Component palette Grouped by purpose. Drag any component into the canvas to create a new slot pre-filled with that component as a FIXED choice. Each palette item pulls its allowed config schema from the components table (M1.3 §6).
4
Canvas — vertical slot stack Top to bottom = render order. Each card represents one slot in the template. Shows slot kind (FIXED / DYNAMIC), slotId, component type + version, and a config preview. Drag to reorder. Click to select.
5
FIXED vs DYNAMIC slots hero, ask-anything, upgrade-teaser are FIXED — admin picked the component, Library only chooses the content. dynamic-rails is DYNAMIC — admin defines what components are allowed and how many; Library picks both the components and the content. See M1.1 §3 for the model.
6
Properties panel — the Component Contract live Every field maps to a key in the ComponentInstance shape (M1.2 §8): config, query, personalization, visibility, copy. Edits autosave to the draft template. Field validation comes from the component's config_schema + query_schema in components (M1.3 §6).
7
Tier visibility When the global monetizationMode is set to open-all (M2.4), this control still saves but does not enforce. When flipped to tier-gated, slots marked tier-restricted are hidden from users outside the allowed tiers. Tier visibility on the upgrade-teaser slot is inverted — it shows only for the listed tiers (i.e., free users).

Interactions & states

Adding a slot

  1. Drag a component from the palette into the canvas at the target position. A drop indicator highlights the insertion point.
  2. On drop, a new FIXED slot is created with the component pre-selected. slotId auto-generates from component type + counter (hero, rail-2, etc.) and is editable in the Properties panel.
  3. Default config values come from the component's schema. Defaults are sensible — the slot renders without further input.

Converting FIXED → DYNAMIC

In the Properties panel, changing kind from FIXED to DYNAMIC converts the slot. The single component selection becomes the first entry in the allows list; admin adds more allowed components, sets minCount/maxCount.

Cloning a template

The Clone button copies the entire template into a new draft with version {original}+1-draft. Cloning preserves all slot configurations and copy. No inheritance — see M1.1 §19 locked decision #28. The cloned template is a fully independent record.

Publishing

  1. Validation runs across the full template:
    • All FIXED slot components must exist in components with status active
    • All DYNAMIC slot allows arrays must contain at least one valid component
    • Every locale-keyed string must have at least en
    • No empty required copy fields
  2. If valid, the draft is promoted to published status. Any previously published version of the same template is archived (kept for rollback).
  3. Existing experiments referencing the template are notified; an admin must explicitly opt the experiment into the new version.

Preview mode

Preview opens the canvas as a rendered page in a side drawer, using the selected "View as" cohort. The render uses Library's compose-page endpoint against the draft template, marked X-Preview-Mode: true so it bypasses cache and includes the full why payload for every slot. Hover any slot in preview to reveal its why reasoning.

Data contract

The Template Builder is a single-page React app at /console/templates/[id]/edit that hits five endpoints:

EndpointPurpose
GET /console/templates/:idLoad the template record + linked component metadata
PATCH /console/templates/:idAutosave a draft (debounced ~10s)
POST /console/templates/:id/publishPromote draft → published
POST /console/templates/:id/cloneClone into new draft
POST /library/compose-pagePower the live preview with overrides.viewAsCohort

Empty states

New template, no slots

Canvas shows a centered message: "Drag a component from the palette to begin." The "Add slot" button is replaced by a larger drop target.

No component selected

Properties panel shows a placeholder: "Select a slot to edit its properties."

Preview with no Library result

Preview panel shows the page rendered with editorial defaults and a banner: "Library returned no personalized content for this cohort — showing trending fallback." This is normal for new cohorts.
