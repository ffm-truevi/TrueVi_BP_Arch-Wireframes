TrueVi Page Builder — Architecture

A specification for the page composition system that powers truevi.tv. Templates are slot-shaped holding tanks. The TrueVi Library fills the slots using AI, behavioral inference, and editorial controls. Pages are rendered per user, per session, per experiment.

The one sentence

Admins define the shape of pages. The TrueVi Library decides what goes inside them. The front-end is a holding tank. Personalization, template selection, content choice, copy, and ordering all happen inside the Library. The renderer is intentionally dumb.

1 · Mental Model

TrueVi.tv is not a CMS in the traditional sense. Traditional CMS systems bind pages to specific content at edit time. TrueVi binds pages to content at render time, per user, per session, based on an evolving behavioral model. Admins do not pick the content; they pick the conditions under which the Library is allowed to pick the content.

This produces four moving parts:

LayerResponsibilityOperated by
Management Console Design templates, define slot constraints, run experiments, publish editorial overrides, manage routes TrueVi internal admins
Page Renderer Read template spec, call Library with envelope, hydrate components, ship HTML/JS to the browser, capture activity events Next.js application
TrueVi Library Select templates, fill slots, infer user state, enforce theological guardrails, allocate experiment variants, cache responses Internal AI & content service
Activity Stream Every impression, click, dwell, completion, save, share, replay is captured and fed back into the inference engine Event pipeline

Why this matters

The renderer never knows whether it is showing crude oil, gasoline, or diesel. It is given a composition spec and renders it. This makes the front-end stable across years of personalization-engine evolution. Replacing the keyword search with a vector DB does not require front-end changes. Replacing the inference model does not require front-end changes. The Library is the brain; the renderer is the body.

2 · The Three Layers

Figure 2.1 — The three layers and the direction of data flow

Boundaries you must not cross

  • The renderer never implements personalization logic. If you find yourself filtering content by faith stage in a React component, stop and move it into the Library.
  • The Console never binds specific content IDs to templates (except via the explicit editorial override system — see §13).
  • The Library never returns raw model output. Every payload it returns conforms to the Component Contract (§4).
  • The Activity Stream never writes to the rendered DOM. It is a one-way feed: browser → events → Library.

3 · Slot-Based Composition

A template is a typed list of slots. Each slot has constraints. The Library is responsible for filling each slot with a concrete component instance at render time, subject to those constraints.

Two slot kinds

KindMeaningLibrary role
FIXED The admin pinned a specific component type. The slot must always render that component. Only the component's content is personalized — not its identity. Fills the slot's query, respecting any admin overrides
DYNAMIC The admin defined constraints only: a list of allowed component types, a min/max count, and zone-level rules. The Library picks which components to render and how many, within those bounds. Decides component identity, count, ordering, and content

Example: HomeTemplate v3

// Stored in the Templates table — see data model M1.3
{
  "templateId": "home-v3",
  "pageType": "home",
  "inputs": [],             // home takes no slot variables
  "slots": [
    {
      "slotId": "hero",
      "kind": "FIXED",
      "component": "Hero-Carousel",
      "config": { "maxSlides": 5, "autoplay": true }
    },
    {
      "slotId": "dynamic-rails",
      "kind": "DYNAMIC",
      "allows": ["ContentRail", "MinistryRail", "SeriesRail"],
      "minCount": 3,
      "maxCount": 10,
      "tierVisibility": ["new", "vine", "plus"]
    },
    {
      "slotId": "ask-anything",
      "kind": "FIXED",
      "component": "AskAnythingBlock",
      "config": { "variant": "hero-card" }
    },
    {
      "slotId": "upgrade-teaser",
      "kind": "FIXED",
      "component": "UpgradeTeaser",
      "tierVisibility": ["new"]   // only free-tier users see this
    }
  ]
}

Decision · No nesting in v1

Templates are a flat vertical stack of slots. Multi-column layouts (e.g., two rails side by side) ship in Phase 2 via a ColumnRow component that holds child slots. The schema reserves this — see §4 — but the v1 builder does not expose it.

4 · The Component Contract

Every renderable component in the system, regardless of where it appears, speaks the same shape. Components are generic — a ContentRail is the same component on Home, Ministry, Subject, and Series pages. Only its parameters differ.

The contract

ComponentInstance {
  // identity
  type:           "ContentRail" | "Hero" | "Grid" | ... 
  instanceId:     "rail-recommended-001"
  version:        "1.4.2"   // templates pin component versions

  // admin-authored, stored on the template
  config:         {
    title:         "Continue Watching",
    subtitle:      null,
    layout:        "horizontal-scroll",
    cardSize:      "medium",
    maxItems:      12,
    showAllLink:   true
  }

  // what the component asks Library for
  query:          {
    contentType:   "video",
    filters:       { faithStage?, lifeStage?, ministry?, tags?, denomination? },
    sort:          "recommended" | "recent" | "trending" | "curated",
    mode:          "auto" | "curated",
    curatedIds?:   ["abc123", "def456"],   // only when mode=curated
    limit:         12
  }

  // passive signals this component is allowed to consume
  personalization: {
    enabled:       true,
    signals:       ["inferredFaithStage", "watchHistory", "exploredSubjects"],
    fallback:      "trending"      // what to do on low confidence
  }

  // gating
  visibility:     {
    tiers:         ["new", "vine", "plus"],
    deviceClasses: ["desktop", "tablet", "mobile"],
    minSessions:   0,
    maxSessions:   null
  }

  // localization
  copy:           {
    en: { title: "Continue Watching", cta: "See all" },
    es: { ... }                                       // reserved, EN-only at launch
  }
}

What the Library returns for each instance

FilledInstance {
  instanceId:    "rail-recommended-001",
  componentType: "ContentRail",
  resolvedConfig: { ... },        // admin config merged with Library overrides
  content:       [ ContentItem, ContentItem, ... ],
  copy:          { title, subtitle, cta },
  why:           {
    reason:      "high_engagement_with_forgiveness_subjects",
    confidence:  0.82,
    signalsUsed: ["watchHistory", "exploredSubjects"]
  },
  experimentVariant: "home-v3-variant-B",
  ttl:           900                  // seconds — see §14
}

The why field

Every Library response includes an explainability payload. It is not surfaced to end users. It powers (a) the "View as User" debugger in the Console, (b) the QA workflow for new experiments, and (c) the analytics pipeline. Optional in v1 production, mandatory in Console preview.

5 · Page Render Sequence

The end-to-end path from a browser request to a fully rendered page:

Figure 5.1 — Sequence diagram, one page render with activity feedback

Notes on the sequence

  • One Library call per page. Not one per component. Library returns the entire composition. This eliminates N-fetch waterfall problems.
  • Activity events are fire-and-forget. The browser does not wait for ingestion. Lost events are acceptable; this is not a financial system.
  • Inference is asynchronous. The warm tier processes the event stream out-of-band. The user's next page reflects the previous page's behavior.
  • TTL governs re-fetch. Same user, same page, within TTL — same composition. See §14.

6 · Passive Personalization

TrueVi does not ask users questions to personalize. There is no onboarding form, no profile wizard, no "tell us about yourself." Every personalization signal is inferred from behavior.

Why this matters

Users lie on profile forms. Users abandon profile forms. Users do not know their own faith stage. Users do not want to declare it even when they do. Behavior tells the truth. Watch what they watch.

The derived signals catalog

DerivedSignals {
  // inferred classifications — produced by the warm-tier model
  inferredFaithStage:    { value: "Seeking",        confidence: 0.61 }
  inferredLifeStage:     { value: "YoungAdult",     confidence: 0.78 }
  inferredLifeEvents:    [
                           { tag: "career-crisis",   confidence: 0.45 },
                           { tag: "spiritual-doubt", confidence: 0.72 }
                         ]

  // behavioral patterns — counters and ratios, updated in hot tier
  watchPatterns:         {
    completionRate:      0.42,
    avgDwellSec:         680,
    formatAffinity:      { sermon: 0.61, devotional: 0.22, teaching: 0.17 },
    preferredSessionMin: 28
  }
  exploredSubjects:      [{ subject: "forgiveness", weight: 0.81 },
                          { subject: "anxiety",     weight: 0.54 }]
  visitedMinistries:     [{ slug: "mercy-hill",     recencyScore: 0.92 }]
  searchHistory:         [{ query: "how to forgive", intent: "emotional-pain" }]
  askAnythingHistory:    [{ topic: "forgiving-father", weight: 0.88 }]
  scriptureEngagement:   [{ ref: "Matthew 18:21",    weight: 0.45 }]

  // session context — computed at request time, not stored
  visitorType:           "first-visit" | "returning" | "regular" | "power"
  sessionContext:        {
    timeOfDay:           "evening",
    dayOfWeek:           "sunday",
    deviceClass:         "desktop",
    referrer:            "google",
    sessionNumber:       7
  }
}

The confidence rule

Every inferred classification carries a confidence score in [0, 1]. The Library uses confidence to decide whether to (a) personalize, (b) fall back to trending, or (c) blend personalization with discovery. Default thresholds: <0.3 = ignore signal, 0.3-0.6 = blend, >0.6 = trust.

7 · Tiered Inference

The user's behavioral profile updates at three different cadences. This preserves both real-time personalization feel and overnight batch guarantees.

TierComputesWhenCost profile
Hot sync Recency scores, last-visited-ministry, current-session subjects, watch progress, save/share state, view counters Inline, on every event, before the event is acknowledged Trivial — counter updates and cache writes
Warm JITP Inferred faith stage, life stage, life-event tags, format affinity, intent classification on search queries Triggered when a threshold is crossed: +10 new events OR +24h stale OR +1 completed video Pay-per-update — only runs when meaningful new signal exists
Cold batch Confidence decay sweeps, cohort recomputation, drift detection, model retraining triggers — no per-user inference Nightly at 02:00 UTC Fixed, scheduled, predictable

Decision · Tiered, not "real-time vs nightly"

Bunker's "fresh every morning" guarantee is satisfied by the Cold tier (decay + cohort recomputation). Brian's "real-time feel" is satisfied by the Hot tier. The Warm tier (the expensive ML inference) only runs when the user has actually done something worth re-classifying. This is measurably cheaper than recomputing every user nightly.

8 · AI Template Selection

Multiple templates can exist for the same page type. The Library decides which template a given user sees on a given request. This is a distinct step from filling the template's slots.

Selection inputs

  • Page type — what kind of page is being rendered (home, ministry, subject, series, search, search-response)
  • User context — derived signals (§6), visitor type, tier
  • Eligible templates — the published templates for this page type
  • Selector rules — admin-authored rules ("first-visit users always see HomeTemplate-Discovery") — see §13
  • Experiment assignments — if this user is in an A/B cohort, the variant template

Selection precedence

  1. If the user is enrolled in a live experiment for this page type, the variant assigned by the experiment wins. Always.
  2. Otherwise, evaluate selector rules in priority order. First matching rule wins.
  3. If no rule matches, the default template for the page type is used.
// Example selector rule, stored in TemplateSelectorRules table
{
  "pageType": "home",
  "priority": 10,
  "name": "first-visit-discovery",
  "condition": { "visitorType": "first-visit" },
  "templateId": "home-discovery-v1"
}

9 · Theological Guardrails

TrueVi has a hard editorial position: no prosperity gospel, no personality-cult content, no theology that compromises the Nicene Creed. These rules are enforced by the Library, not the renderer.

Three layers of enforcement

LayerWhereWhat it does
Ingestion-time Content Library Management System Every piece of content is tagged at ingestion with theological alignment scores, denomination, content warnings, and a pastoral-review status. Prosperity-gospel content is rejected before it ever enters the catalog.
Query-time TrueVi Library Every Library response runs through a guardrail filter that strips any content failing the theological bar — even if an experiment or editorial pin would have surfaced it. Guardrails are always on.
Audit-time Pastoral Board review queue The Pastoral Board reviews flagged edge cases, content scoring changes, and tagging disputes. Their decisions update the ingestion-time and query-time rules.

Console role

The Management Console has a read-only view of active guardrails at /console/guardrails. Edits require Pastoral Board sign-off and are made through a separate workflow. Admins cannot disable guardrails for an experiment.

10 · Cold-Start Strategy

On session #1, the user has zero behavioral history. The Library still needs to produce a useful page. Cold-start uses the request envelope — information available from the HTTP request itself.

Available cold-start signals

  • Referrer — Google query, social link, direct, ministry partner site
  • UTM tags — campaign, source, medium, content, term
  • Landing path — did they hit / or /ministry/mercy-hill?
  • IP geo — country-level only, never street-level
  • Device class — desktop / tablet / mobile
  • Time of day, day of week
  • Browser language

These feed a lightweight cold-start model that produces low-confidence inferences. The Library blends those with editorial defaults and trending content. As the user accumulates 5-10 events, the warm tier kicks in and the inferences sharpen.

11 · A/B Experimentation

Experimentation is a first-class capability, not an add-on. Every template, every slot, and every component can be A/B tested.

Experiment model

Experiment {
  experimentId:    "home-hero-carousel-vs-static"
  pageType:        "home"
  status:          "draft" | "live" | "paused" | "concluded"

  variants: [
    { id: "control", templateId: "home-v3",          allocation: 0.50 },
    { id: "variant-B", templateId: "home-v3-carousel", allocation: 0.50 }
  ]

  // stable assignment — same user always gets same variant for life of experiment
  assignmentKey:   "userId" | "sessionId" | "deviceId"

  primaryKPI:      "watch_completion_rate"
  secondaryKPIs:   ["return_visit_7d", "signup_conversion", "ask_anything_engagement"]

  guardrailKPIs:   ["theological_complaint_rate", "crash_rate"]

  minSampleSize:   2000
  maxDurationDays: 30
  significanceThreshold: 0.95
}

Locked KPIs (from §10 of the Q&A)

  1. Watch completion rate — % of started videos completed past the 80% mark
  2. Return visit — % of users who return within 7 days
  3. Signup conversion — anonymous → registered, attributed to template
  4. Ask Anything engagement — % of sessions that submit at least one query

Assignment guarantees

  • Stable — same user, same variant, for the life of the experiment
  • Sticky across sessions — assignment is persisted, not recomputed
  • Mergeable — when an anonymous user signs up, their pre-signup assignment migrates to the user ID (see §12)
  • Reported on every event — every activity event carries its experiment + variant tags

12 · Identity & Anonymous → Authenticated Merge

Free browsing requires no signup (YouTube TOS compliance — see CLAUDE.md). This means most users start anonymous. Many never sign up. Those who do sign up should not lose their accumulated behavioral history.

Identity hierarchy

IdentityLifetimeUsed for
deviceIdlocalStorage, ~yearsCold-start defaults, device-level analytics
anonymousIdcookie, ~1 yearPre-signup behavioral history accumulator
sessionIdsession cookie, ~hoursPer-session caching, session-level analytics
userIdpermanentAuthenticated user, primary key for all inferences

Merge protocol

  1. Anonymous user accumulates activity_events tagged with their anonymousId.
  2. User signs up. Signup UI displays a consent line: "Carry over your activity from this device so we can keep personalizing for you." (default: checked, can opt out)
  3. On consent, a merge job rewrites anonymous_id → user_id for all events from that anonymousId.
  4. The Warm tier is triggered to recompute inferences from the merged event history.
  5. Experiment assignments tied to the anonymousId are migrated to the userId, preserving cohort integrity.

13 · Editorial Overrides

The Library makes most decisions. Sometimes a human must override it. Three mechanisms exist:

ToolWhat it doesConsole location
Pin Force a specific component instance or content item into a specific slot, for a date range, optionally segment-scoped /console/editorial/pins
Block Prevent specific content from surfacing globally, regardless of inference scores. Used for hot-take pulls, takedown requests, PR-sensitive moments /console/editorial/blocks
Boost Temporarily weight specific ministries, series, or content tags higher in selection scoring. Used for seasonal campaigns (Easter, Christmas), partner pushes, new-content highlights /console/editorial/boosts

Time-windowed and segmentable

All three editorial tools accept a start/end date and an optional audience segment (e.g., visitorType=returning, lifeStage=YoungAdult). Pins and Boosts auto-expire. Blocks persist until explicitly removed.

14 · Caching

Personalization is expensive. Caching is mandatory. But session-level content shifts can disorient users, so caching must be tunable.

Cache hierarchy

LayerScopeDefault TTLInvalidation
Browser memoryper page viewview lifetimenavigation
Edge / CDNper template, anonymous only5 minutestemplate publish
Library composition cacheper (user, pageType, route)15 minutesactivity event, manual flush
Library content cacheper (query signature)1 hourcontent publish
Inference cacheper userwarm-tier output, updated on thresholdwarm-tier run

Open question — Brian flagged

Constant content shifts within a session can disorient users. The composition cache TTL of 15 minutes is a default; the actual right value should be A/B tested. The Library API accepts per-instance TTL overrides for components where stability matters more (e.g., Hero, Continue Watching).

15 · Search Response (Ask Anything)

When a user submits an Ask Anything query, the Library produces a composed response: a curated summary, scripture references, and recommended content. Every response gets a shareable URL.

Storage strategy

Storing raw HTML responses at scale is expensive. Storing composition specs is not. The pattern:

  1. Normalize the query (lowercase, strip filler words, lemmatize).
  2. Compute a semantic hash — an embedding-based hash that groups queries with equivalent meaning. "How do I forgive my father?" and "I can't forgive my dad" map to the same hash.
  3. Cache lookup by hash. On hit, return the cached composition spec.
  4. On miss, generate the response, store the spec (~3-5 KB), return.
  5. Every shared response gets a permanent URL: /answer/[hash].

Storage cost projection

ScaleSpec storageBacking storeCost
1M unique responses~5 GBSQLite or Postgrestrivial
100M unique responses~500 GBCloud SQL~$100/month
1B unique responses~5 TBsharded~$1K/month

Eviction policy

  • Hot — last 30 days OR >5 shares: in-memory + persistent
  • Warm — shared at least once, ever: persistent only
  • Cold — never shared, last accessed >30 days ago: evict; regenerate on demand (same query is the same hash, so regeneration is deterministic and cheap)

16 · Tier-Aware Components

Components can declare which subscription tiers see them. A global setting controls whether tier gating is enforced or bypassed. 

// Component declares tier visibility
visibility: { tiers: ["vine", "plus"] }   // hidden from free tier

// Console global setting overrides
ConsoleSettings.monetizationMode = "open-all"     // pre-launch: all features visible
                                 | "tier-gated"    // post-launch: tiers enforced

The UpgradeTeaser component is the inverse: it appears only for tiers below a target. Its placement and trigger rules are deferred to a later spec — for v1, the Console exposes configurable slots where teasers can appear, and the Library decides whether to fill them based on user signals and experiment state.

17 · Localization

Every user-visible string is locale-keyed from day one. English-only ships at launch. The schema reserves additional locales. 

copy: {
  en: { title: "Continue Watching", cta: "See all" },
  es: { title: "Continuar viendo",  cta: "Ver todo" },     // reserved
  pt: { ... }                                                // reserved
}

The Library returns copy keyed by the user's resolved locale, falling back to en when a translation is missing.

18 · Glossary

TermDefinition
Page TypeA logical kind of page: home, ministry, subject, series, search, search-response. Each route binds to a page type.
TemplateA typed list of slots representing the shape of a page. Multiple templates can exist per page type.
SlotA position in a template that holds a component. Can be FIXED (admin-pinned) or DYNAMIC (Library-filled).
ComponentA generic, reusable UI block (Hero, ContentRail, Grid, etc.). Speaks the Component Contract.
Component InstanceA concrete use of a component in a slot, with config, query, personalization, visibility, and copy.
Filled InstanceWhat the Library returns: a Component Instance with resolved content and copy.
CompositionThe full set of Filled Instances the Library returns for a page render.
Derived SignalA behavioral metric or classification inferred from activity events.
Tier (inference)Hot, Warm, Cold — the three cadences at which derived signals update.
Tier (subscription)New, Vine, Plus — the three subscription levels.
Selector RuleAn admin-authored rule that maps a user context to a specific template.
ExperimentAn A/B test that overrides selector rules for assigned users.
Editorial OverridePin, Block, or Boost — human intervention on Library decisions.
GuardrailA theological filter applied at query time. Always on.
Cold-StartThe first session for a user, where no behavioral history exists.
Semantic HashAn embedding-based hash that groups semantically equivalent Ask Anything queries.

19 · Locked Decisions (from this thread)

  1. Pages are slot-based holding tanks. Library fills slots.
  2. Component palette includes: Hero (Video/Image/Search/AskAnything/Ministry/Subject/Series/Carousel), ContentRail, ContentGrid, FilterChips, MinistryRail, MinistryCard, SeriesRail, SeriesCard, EpisodeList, AskAnythingBlock, ScriptureBlock, AIResponseBlock, PastorCard, ContextBanner, Breadcrumbs, TextBlock, CTABlock, UpgradeTeaser. Phase 2: ColumnRow for nesting.
  3. Personalization is fully passive. No onboarding questions. All signals inferred from behavior.
  4. Multiple templates per page type. Library selects which template to use per request.
  5. Nesting deferred to Phase 2 via ColumnRow.
  6. Tier-aware components, with a Console-global "open-all vs tier-gated" mode toggle.
  7. Live preview in Console with "View as User" selector.
  8. A/B testing is Phase 1, not deferred. Primary KPIs: watch completion, return visit, signup conversion, Ask Anything engagement.
  9. TrueVi Library is one logical service. Vector DB / mock JSON / SQLite are implementation details.
  10. Rails support both auto and curated modes.
  11. Page-level content de-duplication, top-priority slot wins.
  12. Empty/low-confidence fallback is the Library's decision, not the template's.
  13. Tiered inference: Hot (sync), Warm (JITP), Cold (nightly batch).
  14. Templates take typed input parameters (slot variables) — same template serves many URLs.
  15. Per-page-type route map at /console/routes.
  16. Component versioning: templates pin component version; explicit upgrade.
  17. UpgradeTeaser trigger logic deferred — architecture supports configurable placement.
  18. Search Response = a distinct page type from Search results list.
  19. Every Search Response gets a shareable URL via semantic hash.
  20. Same template renders desktop, tablet, mobile via component-level responsive rules.
  21. Anonymous → authenticated history merge with consent at signup.
  22. Cold-start uses request envelope (referrer, UTM, geo, device, time).
  23. Editorial: Pin, Block, Boost. Time-windowed and segmentable.
  24. Inference: tiered (Hot + Warm-JITP + Cold-nightly).
  25. Theological guardrails enforced in the Library, read-only in Console.
  26. Card-level state (saved, in-progress, completed) is universal, not template-controlled.
  27. Locale-keyed copy from day one, English-only at launch.
  28. Templates can be cloned but cannot inherit.
