Global Settings

Console-wide controls that affect every template, every render, every experiment. The monetization mode toggle. Default cache TTLs. The read-only theological guardrails view. Component version pinning. Locale defaults. Routes registry. An audit log of every change.

One sentence

Settings here are system-wide. Changes ripple to every page render on the next cache window. Two of them (monetization mode and theological guardrails) are so consequential they require an explicit confirm step.

Annotated walkthrough

1
Settings nav · grouped by domain Platform, Library, Editorial, Audit. Order matters: most-frequently-changed settings first, irreversible/sensitive settings later.
2
Monetization mode toggle The single most consequential switch in the entire Console. Flipping from open-all to tier-gated immediately hides tier-restricted components from free-tier users on the next cache window. The UI requires a confirm dialog and a "I understand this will change the experience for thousands of users" checkbox.
3
Route registry is mostly auto-managed Developers register routes via code; this screen displays them. Admins can disable a route (returns 404) or change its bound page type — but cannot invent new path patterns from this UI.
4
Inference tuning These knobs change the felt personalization quality. Lower thresholds = more aggressive personalization but more cold-start mistakes. Higher thresholds = safer but slower to feel personalized. Defaults are conservative; A/B test before changing in production.
5
Theological guardrails — display only A read-only mirror of the rules enforced by the Library. Edits require a separate Pastoral Board workflow. This screen exists to make the rules visible to admins and engineers without giving them the ability to disable them. From the architecture (M1.1 §9): guardrails are always on, even for experiments.
6
Audit log on the same page The last 5 changes appear inline so admins notice if a setting changed unexpectedly. Full chronological audit log is on its own subpage.

Confirm-required actions

Two settings require an explicit confirmation step before the change is committed.

Monetization mode

Flipping from open-all to tier-gated hides tier-restricted components from free-tier users — potentially the majority of the audience. The confirm dialog: "This will hide premium components from N free-tier users on the next cache window. Type CONFIRM to proceed."

Route deactivation

Disabling a route causes it to return 404 for all users. Confirm dialog notes inbound traffic from the last 24 hours so the admin can see what they're about to break.

Empty / first-run state

On a fresh install, settings are seeded with the defaults shown above. The first admin to visit the page sees a one-time banner:

Welcome. Your platform is in open-all mode — every feature is visible to every visitor. When you're ready to start monetizing, flip the Platform mode toggle. Until then, the tier-gated settings on each component are saved but not enforced.
