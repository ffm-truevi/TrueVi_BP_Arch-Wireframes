Search Response Template

The Ask Anything answer page. Where a user's open-ended question becomes a curated, scripture-anchored response with recommended content. Every response has a stable, shareable URL via semantic hash. Library may select different response templates based on the question's classified intent.

One sentence

A Search Response is a composed answer — never labeled "AI" — built from a curated text response, scripture anchors, and recommended content the Library pulled together for this specific question.

Copy rule

The word "AI" must never appear in user-facing copy on this page. Position the response as "Personalized Guidance", "What we've gathered", or "For your question". Reinforced in CLAUDE.md and Console copy validation.

Annotated walkthrough

1
Question echo · Hero-AskAnything The user's raw question is displayed back to them in elegant typography. Reinforces that they are seen and understood. Save and Share icons let them bookmark or send the response. Tone label (empathic / theological / practical / scriptural) is the Library's classification — visible for transparency.
2
AIResponseBlock · tone selected by Library The curated text response. The Library picks one of four tones based on the question's classified intent. Notice the eyebrow says "Personalized Guidance" — never "AI." The italicized phrase highlights a theological framing point.
3
Scripture anchor Always present. Library selects the most relevant scripture for the question. For TrueVi+ subscribers, tapping the reference opens the full ScriptureViewer (deferred; legal-cleared translations required — see CLAUDE.md backlog).
4
Recommended content · DYNAMIC slot The Library decides how many rails (typically 1-3) and whether each is a ContentRail or SeriesRail. Long-form questions often surface a series rail. Short, urgent questions surface single-teaching rails sorted by relevance and personalization.
5
Follow-up · context-aware AskAnything The suggestion chips are generated from the original question's intent — they're not generic. If the user clicks a chip, a new Search Response is generated with context.previousResponseId set, enabling multi-turn behavior. Conversation IDs link related responses.
6
Share URL · semantic hash The URL /answer/[hash] is the semantic hash of the normalized query (M1.1 §15). Same question → same hash → same response (until invalidated). Anyone with the URL sees the same answer. Personalization at this layer is intentionally limited — a shared URL must be useful to the recipient too.

Tone-based template variants

Library picks one of four templates per question, based on classified intent. Same six slots; the AIResponseBlock tone + ScriptureBlock selection logic differ.

Template IDToneSelected when
search-response-empathic-v1 empathic Intent = emotional-pain, grief, struggle. Question wording contains emotional markers. Default fallback.
search-response-theological-v1 theological Intent = doctrinal-question, apologetics, "what does scripture say about..."
search-response-practical-v1 practical Intent = how-to, application, "how do I start...", "what should I do..."
search-response-scriptural-v1 scriptural Intent = direct verse query. ScriptureBlock surfaces first; AIResponseBlock contextualizes.

Generation flow

// User submits from AskAnythingBlock on any page
POST /library/ask-anything
{
  "query": "How do I forgive my father?",
  "identity": { ... },
  "request":  { ... }
}

// Library
1. Normalize query → strip filler, lemmatize
2. Compute semantic hash from embedding
3. Cache lookup:
   - HIT  → return stored composition spec (~10-30ms)
   - MISS → generate composition:
            a) Classify intent (empathic/theological/practical/scriptural)
            b) Select Search Response template by intent
            c) Generate AIResponseBlock content with chosen tone
            d) Choose scripture anchor
            e) Query content for recommended rails (personalized)
            f) Generate context-aware follow-up suggestions
            g) Store spec in cache, persist on share
            h) Return composition (~2-4s for fresh generation)

// Renderer
HTTP 200 → navigate to /answer/[responseId]
GET /library/ask-anything?responseId=[id] for the page render
                                  OR
POST /library/compose-page with pageType=search-response, responseId in routeParams

Caching & storage

See M1.1 §15 for the strategy in full. Summary:

TierWhat lives hereStorage
HotLast 30 days OR > 5 sharesRedis-style hot cache + Postgres
WarmShared at least once, persistentPostgres
ColdNever shared, last accessed > 30dEvicted; regenerated on demand (same hash, deterministic)

Storage cost projection (M1.1 §15): 100M unique responses ≈ 500 GB ≈ $100/month on Cloud SQL. Storing composition specs, not rendered HTML, makes this cheap.

Multi-turn conversation

When a user clicks a follow-up suggestion or types a follow-up into the inline AskAnythingBlock, the system creates a related response with a conversationId link. 

POST /library/ask-anything
{
  "query": "What if he's not sorry?",
  "context": {
    "previousResponseId": "a3f9c2d7b1e4...",
    "conversationId":    "conv-2026-05-11-..."     // auto-created on first follow-up
  },
  "identity": { ... },
  "request":  { ... }
}

Library generates a fresh response that references the prior question. Conversations are stored as a separate conversations table joining response IDs in order. Future enhancement: render conversation history as a thread above the current response.

Personalization vs. shareability tension

Important design choice

A shared response URL must produce a useful answer for any viewer — even if they're not the original asker. Personalization is therefore tightly bounded on this page:

  • AIResponseBlock — same content for everyone with the same semantic hash. Not personalized.
  • ScriptureBlock — same verse for everyone. Not personalized.
  • Recommended rails — lightly personalized. The Library re-ranks based on the current viewer's signals, but the pool of candidate content is the same.
  • Follow-up suggestions — same for everyone (derived from the question's intent, not the user).
  • Save/Share UI state — per-viewer (their save state, not the asker's).

Empty & edge states

Invalid responseId (URL tampered)

Returns 404. Renderer shows: "This answer no longer exists. Ask your question again to get a fresh response." with an AskAnythingBlock prefilled with the original query if extractable.

Generation in progress

If a fresh response is being generated (first user to ask this), the renderer shows a skeleton loader for ~2-4s. Subsequent users hit the cache and load instantly.

Library degraded · cannot generate

Returns 503 DEGRADED. Renderer shows: "We can't generate a response right now. Browse teachings on this topic instead:" + a ContentGrid filtered by the original query as keyword search.

Theological guardrail rejected the response

If the generated response or recommended content fails guardrails, Library re-generates with stricter constraints. If it still fails after 3 attempts, returns a hardcoded editorial fallback: "We're not yet ready to answer this with the care it deserves. Here are pastoral teachings we trust on this subject."
