Component Library

The complete catalog of reusable, generic UI blocks that templates compose from. Eighteen distinct components, one with eight variants. Every page in TrueVi is a slot-arrangement of these blocks. This is the dictionary every page template references.

How to read each component spec

Each component is documented in a spec card containing:

  1. Visual mock — a small but realistic rendering in Abyss Teal so designers and PMs recognize what the component is
  2. Config schema — admin-authored knobs (title, layout, maxItems, etc.) — set in the Template Builder Properties panel
  3. Query schema — what the component asks the Library for (only on content-fetching components)
  4. Personalization signals — which derived signals (M1.1 §6) the component may consume
  5. Visibility — tier and device targeting
  6. Variants — sub-types where applicable (e.g., Hero)
  7. Where used — which page templates the component appears in

The Component Contract

Every component implements the ComponentInstance shape from M1.2 §8. Properties on each spec card map directly to keys in that shape. The Library's FilledInstance response (M1.2 §9) is the runtime payload the component renders from.

Trigger logic deferred

Detailed trigger thresholds (after X sessions? after Y video completions?) are deferred to a later spec. For v1, admins place UpgradeTeaser in slots manually and the Library decides whether to fill the slot based on signals.

Phase 2 · ColumnRow

ColumnRow is a layout container that holds 2 or 3 child slots side-by-side. It enables multi-column compositions without nesting at the template level. Reserved in the schema, not built in v1. When ColumnRow ships, every existing template continues to render unchanged.

Planned config

{
  "type": "ColumnRow",
  "config": {
    "columns": 2,
    "gap": 24,
    "breakpointStack": "mobile"     // collapses to vertical on mobile
  },
  "childSlots": [ SlotSpec, SlotSpec ]                // nested slots
}
